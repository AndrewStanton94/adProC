Limitations & assumptions

  • Minimum dimension: 0.1 m
  • Maximum dimension: 3 m
  • Maximum order: 100 boxes
  • The company will not expand beyond cardboard products
  • The company will not add any new options in the forseeable future

Summary of product characteristics

The GUI is defined in the MainFrame class, this consists of code generated by netBeans plus my event handlers and a couple of UI element mutators. I have used events to filter out comboBox elements and toggle checkboxes when the user makes selections to guide the customer to valid combinations, I hope to avoid having the user entering invalid combinations entirely.

The dimensions are validated by DimensionVarifiers, which extend the abstract class InputVarifiers. Errors from the text boxes are displayed imediately in messageLbl on the GUI. Messages are sent about type and range issues.

The box data is stored in child classes of abstract superclass Box. Most of the functions are defined in the superclass, though abstract method getPrice() is overloaded to ensure all requirements are covered without any unneeded checks taking place. Thus improving efficiency. In addition to attributes for the individual box there is a quantity and a BoxMatcher. Placing the quantity in the box seemed more efficient than creating a new collection. The BoxMatcher is a static final attribute.

Determining the box type and creating it is done by BoxMaker. The parameters that effect the type are fed into the static instances of BoxMatcher for each Box sub type, a result of true means the box type has been found.

BoxMatcher has attributes that describe the valid values for the box types and methods that check if the given arguement is valid.

Order is used to store the created boxes in an ArrayList. It has methods to add a new box, return total cost as either a number or a formatted string, and return all boxes as Object[].
